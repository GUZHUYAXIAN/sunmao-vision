## 1. 你需要理解的物理常识

### 1.1 坐标系基准

本项目使用**右手坐标系**，原点在集装箱左下角内壁交汇处：

```
         Y (高)
         │
         │      ╱ Z (深)
         │    ╱
         │  ╱
         │╱──────────── X (宽)
    原点 (0,0,0)

  X 轴 → 集装箱宽度方向（左右）
  Y 轴 → 集装箱高度方向（上下）  ← 重力沿 -Y 方向
  Z 轴 → 集装箱深度方向（前后）
```

**所有尺寸单位统一为毫米 (mm)**，重量统一为千克 (kg)。

> ⚠️ **防坑**: Three.js 默认也使用右手坐标系，Y 轴朝上，这与我们一致。但如果你从其他 3D 引擎（如 Unity 的左手系）移植代码，需要注意坐标转换。

### 1.2 货物的表示方式

每个货物在 3D 空间中用**轴对齐包围盒 (AABB)** 表示：

```
        ┌────────────────┐
       ╱│               ╱│
      ╱ │              ╱ │
     ┌──│─────────────┐  │
     │  │             │  │
     │  │  position   │  │
     │  │  (x,y,z) ●  │  │     position = 包围盒的最小角（左下前角）
     │  └─────────────│──┘
     │ ╱              │ ╱      dimensions = { length(X), width(Z), height(Y) }
     │╱               │╱
     └────────────────┘
```

- `position` 是包围盒**最小角**的坐标（不是中心点！）
- 任何货物的坐标必须满足：`position >= [0,0,0]` 且 `position + dimensions <= [containerWidth, containerHeight, containerDepth]`

> ⚠️ **防坑**: 不同的 3D 库对"position"的定义不同。有些用中心点，有些用底面中心。本项目统一使用**最小角**。在渲染时需要做 offset 转换。

### 1.3 碰撞检测规则

两个 AABB 在任意一个轴上不重叠，它们就不碰撞：

```
碰撞条件（必须同时满足）：
  A.x < B.x + B.length  &&  B.x < A.x + A.length    (X轴重叠)
  A.y < B.y + B.height  &&  B.y < A.y + A.height    (Y轴重叠)
  A.z < B.z + B.width   &&  B.z < A.z + A.width     (Z轴重叠)
```

**边界情况**（面贴面）：

- 两个货物**面贴面**（即共享一个面）**不算碰撞**。
- 例如：`A.x + A.length === B.x` → 它们紧贴但不碰撞 ✅

> ⚠️ **防坑**: 使用 `<` 而非 `<=` 来判断重叠。如果用 `<=` 会把面贴面的合法放置也判为碰撞。

### 1.4 重心稳定性

货物不能"悬浮"——它的底面必须有**足够的支撑**。

```
简化规则：
  1. 货物底面 (Y=0 的面) 必须被下方的支撑面覆盖至少 70%
  2. "下方支撑面" = 集装箱地板 或 其他货物的顶面
  3. 重心投影必须落在支撑面范围内

示意图（侧视）：
      ┌────┐
      │ B  │         ← B 的底面有 ≥70% 在 A 上 → 稳定 ✅
  ┌───┴──┐ │
  │  A   │ │
  └──────┴─┘
  ━━━━━━━━━━━━ 地板

      ┌────┐
      │ B  │         ← B 大部分悬空 → 不稳定 ❌
  ┌─┐ └────┘
  │A│
  └─┘
  ━━━━━━━━━━━━ 地板
```

### 1.5 重量计算

```
单个集装箱：
  净重 (Net Weight) = Σ 所有货物的重量
  扎带重 = Σ 打包带单位重量 × 使用长度 + Σ 拉紧器单位重量 × 使用数量
  毛重 (Gross Weight) = 净重 + 扎带重 + 集装箱皮重 (tareWeight)

全局：
  总净重 = Σ 每个箱的净重
  总毛重 = Σ 每个箱的毛重
```

**集装箱皮重** (Tare Weight)：集装箱空箱时自身的重量。用户在创建集装箱时输入。**如果用户不输入，默认为 0**（此时净重 = 毛重）。

## 2. 数据流转全景图

```
 用户导入 PDF
      │
      ▼
  [外部API] ──→ OBJ 模型文件（存放在 public/models/）
      │
      ▼
  ModelAsset（登记卡）
      │
      ▼  用户手动标定
  ScaleRecord（缩放因子 + 真实尺寸）
      │
      ▼  用户配置属性
  CargoTemplate（产品说明书）
      │
  用户添加集装箱 ──→ Container[]
  用户选择扎带   ──→ LashingConfig
      │
      ▼  组装
  SolveRequest
      │
      ▼  solver 求解（Web Worker 异步）
  SolveResult
      │
      ├──→ Placement[]         → 3D 场景渲染
      ├──→ LashingPlan[]       → 扎带方案展示
      ├──→ Statistics          → 重量/利用率显示
      ├──→ Warning[]           → 警告通知
      └──→ ExportManifest      → Excel 导出
```

**关键原则**：

| 规则               | 解释                                                     |
| ------------------ | -------------------------------------------------------- |
| solver 是纯函数    | `solve(request) → result`，没有副作用、没有状态、没有 IO |
| 前端负责 IO        | 文件读写、用户输入、3D 渲染全部在前端                    |
| contracts 是中间人 | solver 和 web 之间传递的数据必须符合 Zod Schema          |

## 3. 性能优化底线

### 3.1 渲染性能

| 场景                     | 底线要求         |
| ------------------------ | ---------------- |
| 50 个货物 + 1 个集装箱   | 60 FPS           |
| 200 个货物 + 5 个集装箱  | 30 FPS           |
| 500 个货物 + 10 个集装箱 | 15 FPS（可接受） |

### 3.2 求解性能

| 场景                                 | 底线要求              |
| ------------------------------------ | --------------------- |
| 20 种货物，总计 50 件，1 个集装箱    | < 2 秒                |
| 50 种货物，总计 200 件，5 个集装箱   | < 10 秒               |
| 100 种货物，总计 500 件，10 个集装箱 | < 30 秒（显示进度条） |

### 3.3 已确定的优化策略

| 策略                | 说明                                 |
| ------------------- | ------------------------------------ |
| **LOD（细节层次）** | 远距离用低模/包围盒，近距离用高模    |
| **InstancedMesh**   | 同类型货物共享几何体，一次 draw call |
| **按需渲染**        | 场景无变化时不重绘，节省 GPU         |
| **Web Worker**      | solver 在 Worker 线程运行，不阻塞 UI |
| **视锥裁剪**        | Three.js 自动跳过镜头外的物体        |

## 4. 扎带系统说明

### 4.1 为什么需要扎带？

真实的集装箱运输中，货物在海上会受到风浪颠簸。仅靠重力堆叠是不够的——必须用打包带和棘轮拉紧器将货物固定住。

**本系统的扎带方案不是装饰**——它影响求解器的计算：

- 扎带本身有重量（算入毛重）
- 扎带的工作载荷限制了它能固定多大重量的货物
- 扎带的物理路径需要在 3D 场景中可视化

### 4.2 内置6种扎带预设

用户不需要手动输入扎带参数——从预设列表中选择即可。

**打包带（缠绕捆扎货物本身）**：

| 代号          | 材质   | 宽×厚 (mm) | 破断力 (kg) | 场景               |
| ------------- | ------ | ---------- | ----------- | ------------------ |
| `PP-Standard` | 聚丙烯 | 16×0.6     | 250         | 轻型：纸箱、泡沫   |
| `PET-Heavy`   | 聚酯   | 19×1.0     | 680         | 中重型：机械零件   |
| `Steel-Band`  | 钢     | 19×0.8     | 1800        | 超重型：钢材、铸件 |

**棘轮拉紧器（固定到集装箱壁/地板）**：

| 代号         | 带宽 (mm) | 工作载荷 (kg) | 破断力 (kg) | 场景           |
| ------------ | --------- | ------------- | ----------- | -------------- |
| `Ratchet-25` | 25        | 500           | 1500        | 小件固定       |
| `Ratchet-50` | 50        | 2000          | 6000        | 标准（最常用） |
| `Ratchet-75` | 75        | 5000          | 15000       | 重型机械       |

## 5. 常见业务陷阱

以下是外部贡献者最容易踩的坑：

### 陷阱 1: position 是最小角，不是中心

```
❌ 错误理解：position = 物体中心坐标
✅ 正确理解：position = 物体在 X/Y/Z 三个方向上坐标最小的那个角

渲染时需要偏移：
  meshPosition = position + dimensions / 2  （Three.js 的 mesh.position 是中心）
```

### 陷阱 2: 面贴面不是碰撞

```
❌ 错误实现：if (A.x + A.len >= B.x) → 碰撞
✅ 正确实现：if (A.x + A.len > B.x)  → 碰撞（严格大于）
```

### 陷阱 3: 体积利用率不等于面积利用率

```
利用率 = Σ(每个货物的体积) / 集装箱内部体积 × 100%

❌ 不是截面积比
❌ 不是底面积比
✅ 是三维体积比
```

### 陷阱 4: 集装箱皮重默认为 0

```
如果用户创建集装箱时没有填皮重 → tareWeight = 0 → grossWeight = netWeight
❌ 不要因为 tareWeight = 0 就报错或跳过
```

### 陷阱 5: 货物可以嵌套

```
小货物可以放在大货物"内部"（业务上叫"子母件"）。
此时：
  - 小货物的 position 是相对于集装箱原点的绝对坐标（不是相对于大货物的）
  - 碰撞检测时，大货物和小货物之间不做碰撞判定（它们是嵌套关系）
```

### 🔴 难度：地狱级

#### HW-001: 高效 3D 装箱求解算法

**问题**: 在有限时间内（<30秒），对 N 种不同尺寸的货物进行三维装箱，使空间利用率最大化。

**挑战**:

- 这本质上是 NP-Hard 的三维装箱问题 (3D Bin Packing Problem)
- 我们需要的不仅是放进去，还要"像榫卯一样"严丝合缝
- 旧代码中有两种算法（自由空间切割法 + 游标法），但利用率都还有优化空间
- 需要考虑旋转（6种方向）、重力约束、重量约束、载重约束

**参考文献**:

- Dyckhoff et al. "A typology of cutting and packing problems"
- Martello et al. "Three-dimensional bin packing problem"

**旧代码入口**: `legacy_code/SmartContainer/packages/planner-core/src/index.ts` → `buildLoadPlan()`

---

#### HW-002: 自动扎带路径规划

**问题**: 给定一组已放置的货物，自动计算最优的扎带固定路径（打包带绕哪里、拉紧器连到哪个锚点）。

**挑战**:

- 需要考虑扎带的物理路径（不能穿过其他货物）
- 需要保证每个货物都被至少一条扎带固定
- 拉紧器需要连接到集装箱壁/地板上的锚点
- 最小化扎带用量（省成本）同时保证安全

**当前状态**: 尚无旧代码可参考，需要从零设计。

### 🟡 难度：挑战级

#### HW-003: LOD 自动简化策略

**问题**: 将高精度 OBJ 模型（可能数万面片）自动简化为低精度版本，用于远距离渲染。

**挑战**:

- 简化算法需要保持模型轮廓的辨识度
- 需要多级简化（高→中→低→包围盒）
- 需要在浏览器端实现或预计算
- 切换 LOD 级别时不能出现视觉跳变

**技术方向**: 网格简化算法 (Mesh Decimation)、QEM (Quadric Error Metrics)

---

#### HW-004: 3D 拖拽与碰撞吸附

**问题**: 在 3D 场景中用鼠标拖拽货物，实时进行碰撞检测，并在即将碰撞时自动吸附到最近的合法位置。

**挑战**:

- 要把 2D 鼠标移动映射到 3D 空间的正确平面上
- 拖拽过程中需要实时碰撞检测（性能要求高）
- "吸附"效果需要找到最近的合法放置点
- 需要考虑旋转后的碰撞体改变

**旧代码参考**: `legacy_code/20260314-2/SmartContainer-3D/src/lib/geometry.ts`
